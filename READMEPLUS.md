## 重点：原则和减小体积💖
除字幕的sub（sup/psg）外，我们的这套工具还会精简音轨。
字幕的算法和处理，如果同样的语言有sub（图像字幕）和srt/ass（文本字幕），会优先保留文本字幕，剔除图像字幕，因为sbu文件往往比较大，小则几十MB，大则上百MB，这个清理的不算多，重点是多音轨的视频，算法优先原则是保留音轨质量高的同语言，比如8声道的7.1优于6声道的5.1，会优先保留7.1声道，当然你觉得中庸之道更好，要选择5.1，7.1对你没用，那么还是前面说的，扔给AI，自定义就好了。
但这里的核心是剔除什么德语、波兰语、丹麦语（原则是得有English或普通话、粤语），当然说到这里，因为我不懂粤语，所以粤语在有普通话的场景会剔除，实在没有才保留，目前的逻辑是这样。原则还是如果不接受，你可以扔给AI帮你定制。
## 核心重点：流程💡
1. 浏览输入目标文件，可批量，可单个逐个加；
2. 【扫描并预览】主要是先跑一次，给出【计划动作】；
3. 此刻可以人工判断，同时任务中又遗漏跳过的，【开始处理】会自动跳过这个任务；
4. 【扫描并预览】会尽可能在GUI界面输出全部日志：
	- [x] 对于识别的OCR字幕，会提供一部分预览；
	- [x] 音轨的识别是三段式后合并由RapidOCR投票处理，识别主体语言；
	- [x] 结合算法，保留/不保留谁，日志和【计划动作】会展现；
	- [x] 开始时间、结束时间、开销时间、使用网络状况会显示。
5. 【扫描并预览】可以保存记录、导入记录，下次直接【开始处理】；
6. 【扫描并预览】会使用下载流量，缓存内容到本地；
7. 【开始处理】会使用上传流量，毕竟已经缓存了，没必要NAS直读、直写；
8. 由于一定的剔除音轨、字幕可能，上传一般小于下载，会提示任务省了多少GB。

## 核心重点：预缓存机制✅
1. 程序会自动识别是本地片源还是NAS片源：
	- [x] 本地片源绕过缓存设计，直接读源，避免硬盘重复读写；
	- [x] NAS片源强制走缓存，但只缓存1次，我之前的版本能缓存4次😂；
2. 以缓存模式为例，核心是单任务下载1次、上传1次，这是最小使用量，无法避免；
3. 我尝试过全部直接走NAS路径，但是失败了，失败是因为本质还是会缓存；
4. 所以介意写入量的朋友（固态用户）可以走USB挂载NAS：
	- [x] 需要说明，驱动器映射实际还是走UNC，意味着会缓存；
	- [x] 如果基于mklink实际还是走UNC，意味着缓存。
5. 但至少咱们只下载和上传1次，做到最小化读写硬盘。

## 核心重点：速度问题🏃‍♂️‍➡️
1. 第一个任务会先缓存，有等待时间（日志会展现等待多久）；
2. 第一个任务开始扫描，会自动缓存后面的文件：
	- [x] tmp目录下至多产生4个数字缓存目录；
	- [x] 说人话，程序会需要至少4部电影的空间容量。
3. 为什么要预缓存呢？
	- [x] 不是自夸，我迭代这么多版本后，扫描速度很快，单任务20秒~55秒；
	- [x] 按千兆和2.5G算，不预缓存，下个任务会等待，所以必须预缓存。
4. 扫描完成的任务会清理自己的缓存，所以实际【开始处理】会直读NAS（缓存模式），但前面提到了，此刻它实际还是会缓存，并不是真直读，所以会开销上传下载1次的片源容量流量，但这已经是最小化设计了。
5. 能不能直接开始处理呢？这是个好问题，显然是支持的：
	- [x] 先缓存等待检测；
	- [x] 开始检测并缓存下个任务；
	- [x] 根据算法合并当前任务；
	- [x] 清理当前任务缓存空间；
	- [x] 开始下个任务（已缓存好），继续缓存下个任务。
6. 	也就说直接【开始处理】可以少读写一次，少走一次网络。
7. 【扫描并预览】本质是个debug模式，你是可以绕过的。

## 核心重点：AI模型🧠
1. 实际这个我一开始走了很多弯路（但值得😀）：
	- [x] 一开始用Subtitle Edit第三方工具，问题很多，放弃了，更新中没提它；
	- [x] 然后尝试用Tesseract，准确率不太高，最终放弃了；
	- [x] 然后用PaddlePaddle，模型大，速度慢，混合语言不理想；
	- [x] 最后发现还有RapidOCR（OpenVINO）可用，模型小，准确率高速度还快；
2. 语音识别faster-whisper是一直用的，但是前后算法不断优化：
	- [x] 一开始取30秒音轨；
	- [x] 发现准确率太高，降低为10秒；
	- [x] 但刚好可能是空白音轨，或音乐音轨；
	- [x] 然后采用三段式投票，设置中600/1000/1500s体现；
	- [x] 但一个任务要启动3次AI调用，复杂了，于是提取的三段式合并后走AI；
	- [x] 准确率太高，改为三段式5秒合并走AI；
	- [x] faster-whisper支持一段音频中主体语言识别，混合语言的准确性也不错。

## 核心重点：核心工具🛠️
1. 主要是基于FFMPEG和MKVTOOLNIX，这玩意是 mkvtrackfix的核心；
2. 而RapidOCR（OpenVINO）和faster-whisper是灵魂，准确率的fa法宝。

🔥额外说明RapidOCR虽然是走Intel的OpenVINO，但是AMD的处理器也可用用，无非是Intel的处理器速度更快一些💖，同时也正好支持Intel独显如B570、A750、B580等。
🔥实测环境，Windows 10/11，处理器Ultra5 225H：
* Windows 10，极空间Z425虚拟环境，分配6核心/6GB内存；
* Windows 11，笔记本U5 225全核心环境，32GB内存。

🥬实测上述两个环境，速度差不多，也就说最低6核心、6GB就能跑。也就说如果你的NAS性能还不错，实际可用用虚拟机来跑。由于这套本身是Qt+Python，所以你也可用转化为Linux版本，甚至基于Docker来实现，可用找AI帮你搞定。
## 隐藏话题😲
🧠因为我自己的环境是NAS，说人话NAS是linux平台，跑的是Windows平台，那么对于路径、字符标准上也会有问题，Windows支持的字符linux不一定支持。所以我们在【开始处理】中其实加入了一套字符串过滤的设计，同时考虑到电影的命名比较混乱，尤其是名字超长，所以在输出到目标的时候设置了一套算法：

- [ ] 走tmdb取读取电影的中文名、英文名、年份；
- [ ] 对电影文件重命名：
	- [ ] 路径包含中文名，文件名不含中文名；
	- [ ] 路径不包含中文名，文件名包含中文名，提高第三方刮削成功率；
- [ ] 中文名.英文名.年份.分辨率.视频编码.音频编码.flxed.mkv
- [ ] 这样一来，电视剧的问题是不好解决的，所以后面单独做一个电视剧版本；
- [ ] 说人话✅这套工具主要针对电影哦，所以不要去处理电视剧，但可用拿来改；
- [ ] 重命名不涉及修改电影文件夹。

![剪贴板图片](https://github.com/totocan/mkvtrackfix/blob/main/assets/screenshot/2026-07-20%20110221.png)
参考主界面，首要任务是加入目标，也就是【浏览文件夹】和【浏览文件】
两者顾名思义就是批量和单个文件，当然单个文件可以逐步加入多个文件。
加入后实际会自动【收集文件】，那个按钮是多余的，但不排除你要修改识别的格式，所以还是保留，但程序的核心是mp4和mkv，加入的其它格式不一定适用！！！
## 第三步：收集信息✅
![剪贴板图片](https://github.com/totocan/mkvtrackfix/blob/main/assets/screenshot/2026-07-20%20111522.png)
可以看到自动收集的文件列表，会自动识别视频轨道、音轨、字幕轨道信息，视频容量。扫描后会产生【计划动作】，部分如果有问题之类的会自动会【跳过】模式：
- [x] 如果只有一个任务，是跳过模式，那么无法【开始处理】；
- [ ] 如果多个视频任务，含部分跳过模式，可以【开始处理】非跳过的任务。
- [ ] 跳过的任务，可以后期单独处理，日志会输出目录路径，可以复制记录。
## 第四步：扫描并预览✅
扫描开始后，任务栏图标会变化，进入扫描状态模式，处理的任务会变色：
	- [x] 处理中的任务，蓝色背景；
	- [ ] 出现问题的任务，橙色背景；
	- [ ] 严重出错的任务，红色背景；
	- [ ] 处理完的任务，绿色背景。
	
	处理后，日志中会包含任务结束时间，耗时时间，流量消耗等信息。
	流量消耗信息是走网卡的，如果有多网卡可能会失败或报错，请[提报](https://github.com/totocan/mkvtrackfix/issues )。但同时也可以看出来这玩意不是计算硬盘写入和读取的，所以主要作为参考指标。
	🤗同时再说一遍，扫描并预览不是必须项，收集信息后可以直接【开始处理】。
## 	第五步：开始处理✅
开始处理主要基于【扫描并预览】的【计划动作】执行，当然如果没有进行【扫描并预览】同样会基于【扫描并预览】的逻辑实现【计划动作】。
💡主要的差异是【计划动作】产生后直接开始打包为mkv文件，重新按照策略生成新的视频mkv文件，对于NAS来说不用走两次网络流量和硬盘写入量。
✅处理完一个视频任务后，继续下一个视频任务，直到完成所有任务。
✅任务完成后会有简短的报告，开销多少时间，节省多少磁盘空间等信息。
✅节省的空间大小不同，通知消息会不同展示哈，这是彩蛋🧨。
到了，这里，基本就结束了，祝你使用顺利。

# ⚠️ 内测声明

mkvtrackfix 目前处于 活跃开发阶段，功能与逻辑仍在持续迭代中。
