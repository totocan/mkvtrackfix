## 前言：这套工具在解决什么

这套工具的重点是优化、修正原始 mp4 / mkv 视频文件的标签。比如我们随便打开一个 mkvtoolnix 浏览的视频流：

![mkvtoolnix 原始音轨流](assets/screenshot/01-mkvtoolnix-stream.png)

初看上面这个没啥问题，ID 默认标签都对，语言也符合标准。但名称部分的音频轨道就不太好了——直接基于视频名命名，部分播放器显示不了那么长，徒增负担。

![优化后的音轨标签对比](assets/screenshot/02-after-optimize.png)

优化后是上面这样的：首先语言基于 AI 用完整片段算法去判定（因为有些是错的，尤其大量 `und` 标签）；其次名称方面我们结合了「语言 + 音轨编码」，更直观。

![语言优先级设置](assets/screenshot/03-lang-priority.png)

针对包含普通话、中文语言的，我们默认**国语优先**原则：简中英双语 ＞ 简体中文 ＞ English。当然如果只有 English 和繁体中文，繁体中文也会保留。

![字幕处理选项](assets/screenshot/04-subtitle-options.png)

我个人有我的需求，所以如上图默认是：

- [x] 移除繁体中文字幕
- [ ] 如果需要保留，你可以去掉这个勾选

我个人的看法是繁体中文没意思，反正没有的话我也可以去刮削一个简体中文或双语的。但这里好歹给了大家选择，如果这个选项没用，可以提报，因为没实测。

还有一种场景，比如以 NF 片源为例，会包含大量的多国语言：

![NF 多语言字幕列表](assets/screenshot/05-multilang-nf.png)

虽然这种字幕多数为文本字幕，单个语言往往只有 100KB 左右，不占空间。但部分电视、部分播放器默认不能优先选简体中文，选择时会变得异常灾难——列出来的太多了，无从下手，虽然标签完全符合规范。尤其是用遥控器选择时，会非常麻烦。所以这套工具会对这些下手，最终精简化。

所以如果你想保留、仅仅是识别和优化错误的标签，建议你基于这套工具直接打包下载，然后甩给 AI（如前面介绍的 Gemini 或 WorkBuddy），让 AI 给你定制需求、重构部分设计逻辑，就可以按需使用了。当然你也可以开源分享出来。

## 重点：原则与减小体积

除字幕的 sub（sup / pgs）外，这套工具还会精简音轨。

字幕的算法和处理：如果同一语言同时有 sub（图像字幕）和 srt / ass（文本字幕），会优先保留文本字幕、剔除图像字幕。因为 sup 文件往往比较大，小则几十 MB，大则上百 MB。不过这部分清理的量不算多，**重点是多音轨的视频**——算法优先原则是保留同语言中质量更高的音轨，比如 8 声道的 7.1 优于 6 声道的 5.1，会优先保留 7.1。当然你觉得中庸之道更好、想选 5.1，7.1 对你没用，那么还是前面说的：扔给 AI，自定义就好。

这里的核心是剔除德语、波兰语、丹麦语等（原则是得有 English 或普通话、粤语）。说到这里，因为我不懂粤语，所以粤语在有普通话的场景会被剔除，实在没有才保留，目前的逻辑是这样。原则还是那句：如果不接受，你可以扔给 AI 帮你定制。

## 核心重点：流程

1. 浏览输入目标文件，可批量，也可单个逐个加；
2. **【扫描并预览】** 主要是先跑一次，给出**【计划动作】**；
3. 此刻可以人工判断；同时任务中若有遗漏跳过的，**【开始处理】** 会自动跳过这个任务；
4. **【扫描并预览】** 会尽可能在 GUI 界面输出全部日志：
   - [x] 对于识别的 OCR 字幕，会提供一部分预览；
   - [x] 音轨的识别是三段式后合并，由 RapidOCR 投票处理，识别主体语言；
   - [x] 结合算法，保留 / 不保留谁，日志和**【计划动作】**会展现；
   - [x] 开始时间、结束时间、开销时间、使用网络状况会显示。
5. **【扫描并预览】** 可以保存记录、导入记录，下次直接**【开始处理】**；
6. **【扫描并预览】** 会使用下载流量，缓存内容到本地；
7. **【开始处理】** 会使用上传流量，毕竟已经缓存了，没必要 NAS 直读、直写；
8. 由于一定可能剔除音轨、字幕，上传一般小于下载，会提示任务省了多少 GB。

## 核心重点：预缓存机制

1. 程序会自动识别是本地片源还是 NAS 片源：
   - [x] 本地片源绕过缓存设计，直接读源，避免硬盘重复读写；
   - [x] NAS 片源强制走缓存，但只缓存 **1 次**（我之前的版本能缓存 4 次 😂，现已优化为滑动窗口）。
2. 以缓存模式为例，核心是单任务下载 1 次、上传 1 次，这是最小使用量，无法避免；
3. 我尝试过全部直接走 NAS 路径，但是失败了——失败是因为本质还是会缓存；
4. 所以介意写入量的朋友（固态用户）可以走 USB 挂载 NAS：
   - [x] 需要说明，驱动器映射实际还是走 UNC，意味着会缓存；
   - [x] 如果基于 `mklink` 实际还是走 UNC，意味着会缓存。
5. 但至少咱们只下载和上传 1 次，做到最小化读写硬盘。

## 核心重点：速度问题

1. 第一个任务会先缓存，有等待时间（日志会展现等待多久）；
2. 第一个任务开始扫描，会自动缓存后面的文件：
   - [x] `tmp` 目录下至多产生 **3 个**数字缓存目录（当前任务 + 2 个向前预取）；
   - [x] 说人话，程序会需要至少 **3 部电影** 的临时空间容量。
3. 为什么要预缓存呢？
   - [x] 不是自夸，迭代这么多版本后，扫描速度很快，单任务 20 秒 ~ 55 秒；
   - [x] 按千兆和 2.5G 算，不预缓存，下个任务会等待，所以必须预缓存。
4. 扫描完成的任务会清理自己的缓存，所以**【开始处理】** 会直写 NAS（缓存模式）；但前面提到了，此刻它实际还是会缓存，并不是真直读，所以会开销上传 + 下载 1 次的片源容量流量，但这已经是最小化设计了。
5. 能不能直接开始处理呢？这是个好问题，显然是支持的：
   - [x] 先缓存等待检测；
   - [x] 开始检测并缓存下个任务；
   - [x] 根据算法合并当前任务；
   - [x] 清理当前任务缓存空间；
   - [x] 开始下个任务（已缓存好），继续缓存下个任务。
6. 也就是说直接**【开始处理】** 可以少读写一次，少走一次网络。
7. **【扫描并预览】** 本质是个 debug 模式，你是可以绕过的。

## 核心重点：AI 模型

1. 实际这个我一开始走了很多弯路（但值得 😀）：
   - [x] 一开始用 Subtitle Edit 第三方工具，问题很多，放弃了，更新中没提它；
   - [x] 然后尝试用 Tesseract，准确率不太高，最终放弃了；
   - [x] 然后用 PaddlePaddle，模型大、速度慢，混合语言不理想；
   - [x] 最后发现还有 RapidOCR（OpenVINO）可用，模型小、准确率高、速度还快；
2. 语音识别 faster-whisper 是一直用的，但前后算法不断优化：
   - [x] 一开始取 30 秒音轨；
   - [x] 发现准确率太高，降低为 10 秒；
   - [x] 但刚好可能是空白音轨，或音乐音轨；
   - [x] 然后采用三段式投票，设置中 600 / 1000 / 1500s 体现；
   - [x] 但一个任务要启动 3 次 AI 调用，复杂了，于是提取的三段式合并后走 AI；
   - [x] 准确率太高，改为三段式 5 秒合并走 AI；
   - [x] faster-whisper 支持一段音频中主体语言识别，混合语言的准确性也不错。

## 核心重点：核心工具

1. 主要是基于 FFMPEG 和 MKVTOOLNIX，这玩意是 mkvtrackfix 的核心；
2. 而 RapidOCR（OpenVINO）和 faster-whisper 是灵魂，准确率的法宝。

🔥 额外说明：RapidOCR 虽然走 Intel 的 OpenVINO，但 AMD 的处理器也能用，无非是 Intel 的处理器速度更快一些 💖，同时也正好支持 Intel 独显如 B570、A750、B580 等。

🔥 实测环境，Windows 10 / 11，处理器 Ultra5 225H：

- Windows 10，极空间 Z425 虚拟环境，分配 6 核心 / 6GB 内存；
- Windows 11，笔记本 U5 225 全核心环境，32GB 内存。

🥬 实测上述两个环境，速度差不多。也就是说最低 6 核心、6GB 就能跑。也就是说如果你的 NAS 性能还不错，实际可以用虚拟机来跑。由于这套本身是 Qt + Python，所以你也可以转化为 Linux 版本，甚至基于 Docker 来实现，可以找 AI 帮你搞定。

## 隐藏话题：NAS 路径与重命名

🧠 因为我自己的环境是 NAS，说人话 NAS 是 linux 平台、跑的是 Windows 平台，那么对于路径、字符标准上也会有问题——Windows 支持的字符 Linux 不一定支持。所以我们在**【开始处理】** 中其实加入了一套字符串过滤的设计；同时考虑到电影的命名比较混乱，尤其是名字超长，所以在输出到目标时设置了一套算法：

- [ ] 走 TMDB 去读取电影的中文名、英文名、年份；
- [ ] 对电影文件重命名：
  - [ ] 路径包含中文名、文件名不含中文名；
  - [ ] 路径不包含中文名、文件名包含中文名，提高第三方刮削成功率；
- [ ] 命名格式：`中文名.英文名.年份.分辨率.视频编码.音频编码.flxed.mkv`
- [ ] 这样一来，电视剧的问题是不好解决的，所以后面单独做一个电视剧版本；
- [ ] 说人话 ✅ 这套工具主要针对**电影**哦，所以不要去处理电视剧，但你可以拿来改；
- [ ] 重命名不涉及修改电影文件夹。

# 操作流程（怎么用？）

前期没有 Releases 版本，所以 Code 部分打包下载：

![GitHub Code 下载入口](assets/screenshot/06-code-download.png)

解压后，可以看到 mkvtrackfix 目录，第一次先跑批处理构建环境：

![构建环境批处理](assets/screenshot/07-build-env.png)

它会很友好地帮你把所有环境如 Python、大模型、工具爬下来，模型优先走魔塔 🪄，魔塔不行抱抱脸 🤗，实在不行自己想办法。但后期 Releases 出来会有额外的环境包。

环境好了怎么用？有两个 bat 批处理文件：

![run / tray 批处理文件](assets/screenshot/08-bat-files.png)

虽然选择 run.bat 可以，但是少了 tray.bat 任务栏监控，推荐使用 tray.bat 启动：

![托盘墨镜图标](assets/screenshot/09-tray-icon.png)

右下角的墨镜图标（它实际会根据状态变化），选择**【运行程序】** 启动，窗口中一定会有 Python 的命令行窗口，**千万不要去关闭**，关闭等于退出哦。

![正常主界面](assets/screenshot/10-main-ui.png)

正常的界面如上。

## 第一步要做什么？

- [x] 左上角**【设置】** 打开看看，了解下选项，一般情况不改；
- [ ] 根据需求可以修改一些参数，比如：

![设置 - 音轨采集起点](assets/screenshot/11-settings-audio.png)

可以看到音轨采集的起点，用英文逗号分隔，采样时间默认还是 10 秒，可以最低 5 秒。然后下面还有个针对 RapidOCR 的尝试次数，这个看起来很大，实际可以不管——为啥呢？一般第一次不行，第二次就行了，所以实际不会每次都跑 4 次哦。

但是上面的 TMDB 联网是一定要开的哦，不开很多问题就出来了，准确是第一要素。

🤗 关于这点，AI 被我骂了很多次了，它老想忽略我 AI 检测、联网查询的意义。

![设置 - 字体与字号](assets/screenshot/12-settings-font.png)

然后字体和字号你可以根据自己的喜欢调整。

最后的**【调试模式】** 是不会清理 `tmp` 目录下扫描和执行的缓存的，方便你调试。但是关闭程序本身后还是会清理掉哦。也就是说要调试，得保持程序运行。

第三方工具和 AI 模型会有个独立的详细日志，在 `logs` 文件夹中，必要可以甩给 AI 处理。

## 第二步：加入目标文件

![主界面 - 加入目标](assets/screenshot/10-main-ui.png)

参考主界面，首要任务是加入目标，也就是**【浏览文件夹】** 和**【浏览文件】**。

两者顾名思义就是批量和单个文件，当然单个文件可以逐步加入多个文件。

加入后实际会自动**【收集文件】**，那个按钮是多余的，但不排除你要修改识别的格式，所以还是保留。但程序的核心是 mp4 和 mkv，加入的其它格式不一定适用！！！

## 第三步：收集信息

![自动收集的文件列表](assets/screenshot/13-file-list.png)

可以看到自动收集的文件列表，会自动识别视频轨道、音轨、字幕轨道信息，以及视频容量。扫描后会产生**【计划动作】**，部分如果有问题之类的会自动进入**【跳过】** 模式：

- [x] 如果只有一个任务、且是跳过模式，那么无法**【开始处理】**；
- [ ] 如果多个视频任务、含部分跳过模式，可以**【开始处理】** 非跳过的任务；
- [ ] 跳过的任务，可以后期单独处理，日志会输出目录路径，可以复制记录。

## 第四步：扫描并预览

扫描开始后，任务栏图标会变化，进入扫描状态模式，处理的任务会变色：

- [x] 处理中的任务：蓝色背景；
- [ ] 出现问题的任务：橙色背景；
- [ ] 严重出错的任务：红色背景；
- [ ] 处理完的任务：绿色背景。

处理后，日志中会包含任务结束时间、耗时时间、流量消耗等信息。

流量消耗信息是走网卡的，如果有多网卡可能会失败或报错，请[提报](https://github.com/totocan/mkvtrackfix/issues)。但同时也可以看出来这玩意不是计算硬盘写入和读取的，所以主要作为参考指标。

🤗 同时再说一遍：**扫描并预览不是必须项**，收集信息后可以直接**【开始处理】**。

## 第五步：开始处理

开始处理主要基于**【扫描并预览】** 的**【计划动作】** 执行；当然如果没有进行**【扫描并预览】**，同样会基于**【扫描并预览】** 的逻辑生成**【计划动作】**。

💡 主要的差异是**【计划动作】** 产生后直接开始打包为 mkv 文件，重新按照策略生成新的视频 mkv 文件，对于 NAS 来说不用走两次网络流量和硬盘写入量。

✅ 处理完一个视频任务后，继续下一个视频任务，直到完成所有任务。
✅ 任务完成后会有简短的报告：开销多少时间、节省多少磁盘空间等信息。
✅ 节省的空间大小不同，通知消息会不同展示哈，这是彩蛋 🧨。

到了这里，基本就结束了，祝你使用顺利。

# ⚠️ 内测声明

mkvtrackfix 目前处于**活跃开发阶段**，功能与逻辑仍在持续迭代中。

虽然工具设计上严格遵循**非破坏式输出** 原则——所有处理均输出为 `.flxed.mkv` 或 smart_rename 命名的新文件，绝不修改原始媒体文件——但考虑到涉及 AI 识别、轨道编辑、mkvmerge 重封装等复杂操作，强烈建议：

- 首次使用前先在**测试目录** 中对少量文件反复验证，确认策略符合预期
- 确认无误后再对正式媒体库执行批量处理
- 重要数据请提前备份

因使用本工具导致的任何数据丢失、文件损坏或其他损失，开发者**不承担责任**。请根据自身情况评估风险后使用。
